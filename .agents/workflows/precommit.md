# Precommit Workflow

コミット前に実行するチェックの標準手順。AI エージェントが「/precommit」等のコマンドから実行する共通ロジックを定義する。

## ゴール

コミット対象のコード・ドキュメント・設定ファイルが以下を満たすことを確認する:

- **整形** が適用済み（フォーマッタ）
- **静的解析** に違反がない（リンター / 型チェック）
- **テスト** がグリーン（ユニット / 統合）
- **ビルド** が成功する（該当プロジェクトの場合）
- **依存スナップショット** が最新（DB マイグレーション、ロックファイル、generated コード等）

どのコマンドで上記を満たすかは **プロジェクト依存**。実行コマンドは `CONTEXT.md` または `.agents/rules/*.md` に記載すること。

## 実行手順

1. プロジェクトの precommit コマンドを実行する。
   - Node/TS プロジェクトの例: `task precommit` / `npm run precommit` / `pnpm precommit`
   - Python プロジェクトの例: `task precommit` / `ruff check && pytest`
   - Go プロジェクトの例: `go vet ./... && go test ./...`
   - Rust プロジェクトの例: `cargo clippy && cargo test`
   - 正しいコマンドは `CONTEXT.md` の「開発コマンド」節を参照

2. エラーが発生した場合、原因別に対応する:
   - **整形違反**: フォーマッタで自動修正して再実行
   - **lint 違反**: ルール違反を修正（`eslint --fix` 等の自動修正が効く場合は利用）
   - **型エラー**: 根本原因を特定して修正。型注釈だけ緩めて回避しない
   - **テスト失敗**: プロダクションコードを修正する。**テストを壊して通さない**
   - **ビルド失敗**: 依存関係・設定ファイル・import の整合性を確認
   - **スナップショット不整合**: 更新手順を `CONTEXT.md` または `.agents/rules/` で確認

3. 全チェックがパスするまで 1〜2 を繰り返す。

4. 最終確認として `git status` を表示し、コミット対象が意図通りか確認する。

## ガードレール

- **テストを通すためにテスト自体を変更しない**。テストが失敗する場合、修正すべきはプロダクションコードである
- **警告を抑止しない**。警告は根本原因を修正して解消する
- **CI でしか失敗しないチェック** がある場合は CI ログを参照し、ローカルで再現できるよう手順を `.agents/rules/` に追記する
- スキーマ定義を変更した場合、関連するマイグレーション / generated コード / 型定義が同期されているか確認する（詳細はプロジェクト側の規約に従う）

## 呼び出し方

- Claude Code: `/precommit` スラッシュコマンド（`.claude/commands/precommit.md` が本 workflow を参照する）
- Codex CLI / Cursor: 直接本ファイルを参照して手順を実行する（`@.agents/workflows/precommit.md`）
