# Fix CI Workflow

CI の失敗を特定し、TDD サイクルで修正するための標準手順。

## ゴール

- CI ジョブの赤を根本原因から解消する
- 再発を防ぐためのリグレッションテストを追加する
- ローカルで再現可能な状態にしてからコミット・プッシュする

## 前提

- GitHub Actions / GitLab CI 等の CI システムが有効
- `gh` CLI（GitHub の場合）または同等のログ取得手段が利用可能
- ローカル環境で該当プロジェクトのテストがグリーンに通る状態

## 実行手順

1. 失敗しているチェックを特定する:
   - GitHub: `gh pr checks`（または `gh run view <run-id>`）
   - GitLab: `glab ci list` 等

2. 失敗ログを取得し、根本原因を分析する:
   - エラーメッセージ、スタックトレース、差分を確認
   - ローカル環境との差異（環境変数、OS、依存バージョン、DB 状態）を疑う
   - 直近のコミットで導入された変更との因果関係を確認

3. TDD サイクルで修正する:
   - **RED**: 失敗を再現するテストを作成または特定。既存テストが失敗しているならそれを再利用
   - **GREEN**: テストが通る最小限の修正をプロダクションコードに加える
   - **REFACTOR**: 可読性・一貫性のために整える。テストは動いたまま

4. ローカルで `precommit workflow` を実行し全体の整合性を確認する（`@.agents/workflows/precommit.md`）。

5. `fix:` プレフィックスで論理単位ごとにコミットする:

       fix(area): 修正内容を 1 行で要約

       - 根本原因
       - 取った対策
       - 追加 / 更新したテスト

6. プッシュして CI の再実行を待つ。再度失敗した場合は 1 から繰り返す。

## ガードレール

- **テストを通すためにテスト自体を変更しない**。修正すべきはプロダクションコード
- **フレーク（間欠失敗）は即座に原因分析する**。再実行で通ったとしてもテストの非決定性を放置しない
- **CI 専用の hack（env 決め打ち、特殊スクリプト）を書く前にローカル再現を試みる**
- 原因不明のまま「通ったっぽい」状態で close しない。Decision Log / Surprises & Discoveries に所見を残す

## 呼び出し方

- Claude Code: `/fix-ci [引数]` スラッシュコマンド
- Codex CLI / Cursor: `@.agents/workflows/fix-ci.md` を参照して実行
