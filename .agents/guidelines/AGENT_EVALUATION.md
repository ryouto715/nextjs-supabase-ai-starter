# エージェント評価・品質保証ガイドライン

> **目的**: エージェントが生成したコード・成果物の品質を評価し、継続的に改善するための基準とプロセスを定義する。

---

## 1. 評価の対象と目的

### 1.1 何を評価するか

| 対象 | 評価観点 | 評価タイミング |
|------|----------|---------------|
| **REQUIREMENTS.md** | 完全性、明確性、実現可能性 | フェーズ1完了時 |
| **ExecPlan** | タスク分解の適切さ、見積もり精度 | 実装開始前 |
| **生成コード** | 品質、保守性、セキュリティ | PR レビュー時 |
| **テストコード** | カバレッジ、テストケースの妥当性 | PR レビュー時 |
| **ワークフロー全体** | 効率性、成功率、リードタイム | 定期レビュー |

### 1.2 評価しないこと

- エージェントの内部思考プロセス（観測不可）
- トークン消費量の最適化（制御困難）
- 生成速度（インフラ依存）

## 2. REQUIREMENTS.md の評価

### 2.1 チェックリスト

**完全性**:
- [ ] ビジョンとゴールが明確に記述されている
- [ ] ターゲットユーザーが具体的に定義されている
- [ ] 機能要件に受け入れ基準が設定されている
- [ ] 技術スタックの選定理由が記載されている
- [ ] スコープ（MVP / 将来）が明確に分離されている

**明確性**:
- [ ] 曖昧な表現（「など」「適切に」）が排除されている
- [ ] 各機能の優先度が明示されている
- [ ] 専門用語が用語集で定義されている

**実現可能性**:
- [ ] 技術的に実装可能な範囲である
- [ ] 選定した技術スタックで要件を満たせる
- [ ] 外部依存（API、サービス）が明確

**調査の質**:
- [ ] 市場・競合調査が実施されている
- [ ] 調査ソースが明記されている
- [ ] 調査結果が機能設計に反映されている

### 2.2 評価スコア

| 項目 | 基準 | 配点 |
|------|------|------|
| ゴール明確性 | ユーザーの目的が 1-2 文で説明できる | 20 |
| 機能完全性 | MVP に必要な機能が網羅されている | 25 |
| 受け入れ基準 | 全機能に検証可能な基準がある | 20 |
| 技術選定 | 選定理由が目的と整合している | 15 |
| 調査品質 | 競合分析が意思決定に反映されている | 20 |

**合格ライン**: 80 点以上

## 3. 生成コードの評価

### 3.1 自動チェック

**必須（CI で実行）**:
```yaml
# .github/workflows/code-quality.yml
name: Code Quality

on: [push, pull_request]

jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      - name: Lint
        run: npm run lint

      - name: Type check
        run: npm run type-check

      - name: Test
        run: npm test -- --coverage

      - name: Check coverage threshold
        run: |
          COVERAGE=$(cat coverage/coverage-summary.json | jq '.total.lines.pct')
          if (( $(echo "$COVERAGE < 70" | bc -l) )); then
            echo "Coverage $COVERAGE% is below 70% threshold"
            exit 1
          fi
```

**推奨チェック**:
- ESLint / Prettier（コードスタイル）
- TypeScript strict mode（型安全性）
- Vitest / Jest（テスト）
- Playwright / Cypress（E2E）
- npm audit / Snyk（セキュリティ）

### 3.2 コードレビューチェックリスト

**構造・設計**:
- [ ] 単一責任の原則に従っている
- [ ] 適切な抽象化レベルである（過剰でも不足でもない）
- [ ] 依存関係が適切に管理されている
- [ ] 循環依存がない

**可読性**:
- [ ] 命名が意図を表している
- [ ] 複雑な処理にコメントがある
- [ ] マジックナンバーが定数化されている
- [ ] ネストが深すぎない（3 レベル以内）

**エラーハンドリング**:
- [ ] すべての外部呼び出しにエラー処理がある
- [ ] 適切なエラーメッセージを返している
- [ ] エラーがログに記録されている
- [ ] ユーザーに生のエラーを見せていない

**セキュリティ**:
- [ ] ユーザー入力がサニタイズされている
- [ ] SQL インジェクション対策がある
- [ ] XSS 対策がある
- [ ] 認証・認可が適切に実装されている
- [ ] シークレットがハードコードされていない

**パフォーマンス**:
- [ ] N+1 クエリがない
- [ ] 適切なインデックスが設定されている
- [ ] 不要な再レンダリングがない（React）
- [ ] メモ化が適切に使用されている

### 3.3 テストコードの評価

| 項目 | 基準 |
|------|------|
| カバレッジ | ビジネスロジック: 80%+、全体: 70%+ |
| テスト種別 | 単体・統合・E2E のバランス |
| エッジケース | 境界値、null、空配列等のテスト |
| モック使用 | 外部依存の適切なモック化 |
| テスト名 | 何をテストしているか明確 |

## 4. ワークフロー全体の評価

### 4.1 メトリクス

**成功率メトリクス**:

| 指標 | 計算方法 | 目標 |
|------|----------|------|
| 実装成功率 | 完了タスク / 開始タスク | 95%+ |
| 初回 PR 承認率 | 修正なしでマージされた PR / 全 PR | 80%+ |
| テスト成功率 | テスト通過 / テスト実行 | 100% |

**品質メトリクス**:

| 指標 | 計算方法 | 目標 |
|------|----------|------|
| バグ発生率 | リリース後のバグ報告数 / 機能数 | 0.1 未満 |
| コードカバレッジ | テストでカバーされた行 / 全行 | 70%+ |
| 技術的負債 | TODO / FIXME コメント数 | 減少傾向 |

**効率性メトリクス**:

| 指標 | 説明 |
|------|------|
| リードタイム | 要件確定から PR マージまでの時間 |
| サイクルタイム | 実装開始から PR マージまでの時間 |
| 手戻り回数 | PR での修正依頼回数 |

### 4.2 定期レビュー

**週次**:
- 完了した PR のレビュー品質確認
- 発生したバグの根本原因分析
- AGENTS.md / ガイドラインの改善点特定

**月次**:
- メトリクスの集計・トレンド分析
- ワークフローのボトルネック特定
- ガイドラインの更新

## 5. 評価プロセス

### 5.1 PR レビューフロー

```
[PR 作成]
    ↓
[自動チェック実行]
    ├─ Lint ✓
    ├─ Type Check ✓
    ├─ Test ✓
    └─ Coverage ✓
    ↓
[自動チェック通過？]
    ├─ No → 修正依頼（エージェント or 人間）
    └─ Yes ↓
[人間によるコードレビュー]
    ├─ チェックリスト確認
    ├─ ビジネスロジック確認
    └─ セキュリティ確認
    ↓
[承認 / 修正依頼]
    ↓
[マージ]
```

### 5.2 問題発見時の対応

**即座に修正**:
- セキュリティ脆弱性
- 明らかなバグ
- 生のエラー露出

**ガイドライン更新を検討**:
- 同じパターンの問題が繰り返し発生
- レビューで毎回指摘する項目
- 新しいベストプラクティスの発見

## 6. 品質改善サイクル

### 6.1 PDCA サイクル

```
Plan（計画）
├─ 現状のメトリクス確認
├─ 改善目標の設定
└─ 具体的な施策の立案
    ↓
Do（実行）
├─ ガイドライン更新
├─ 新しいチェックの追加
└─ エージェントへの指示改善
    ↓
Check（評価）
├─ メトリクスの変化確認
├─ レビュー品質の確認
└─ フィードバック収集
    ↓
Act（改善）
├─ 効果のあった施策の定着
├─ 効果のなかった施策の見直し
└─ 次サイクルの計画
```

### 6.2 ガイドライン改善のトリガー

以下の場合にガイドラインを更新する：

| トリガー | 対応 |
|---------|------|
| 同じ指摘が 3 回以上発生 | チェックリストに追加 |
| セキュリティインシデント | セキュリティポリシー強化 |
| 新技術・パターンの採用 | 対応するガイドライン追加 |
| 開発効率の低下 | プロセスの見直し |

## 7. フィードバックループ

### 7.1 エージェントへのフィードバック

PR レビューでの指摘事項は、以下の形式で AGENTS.md に反映を検討：

```markdown
## 避けるべきパターン

### [パターン名]
- **問題**: [何が問題か]
- **例**: [具体的なコード例]
- **対策**: [正しいアプローチ]
```

### 7.2 継続的な学習

1. **Decision Log の活用**: 過去の意思決定を参照し、一貫性を保つ
2. **エラーパターンの蓄積**: 発生したエラーとその解決策を記録
3. **ベストプラクティスの共有**: 良いコード例を AGENTS.md に追加

## 8. チェックリストサマリー

### フェーズ1（REQUIREMENTS.md）完了時

- [ ] ビジョン・ゴールが明確
- [ ] 機能要件に受け入れ基準がある
- [ ] 技術スタック選定に根拠がある
- [ ] 市場調査が実施されている
- [ ] スコープが明確

### フェーズ2（PR）レビュー時

- [ ] 自動チェックがすべて通過
- [ ] コードが REQUIREMENTS.md を満たしている
- [ ] テストカバレッジが基準を満たしている
- [ ] エラーハンドリングが適切
- [ ] セキュリティチェックリストを通過
- [ ] パフォーマンス上の問題がない

### リリース前

- [ ] E2E テストが通過
- [ ] 手動での動作確認完了
- [ ] ドキュメントが更新されている
- [ ] 必要な環境変数が設定されている
