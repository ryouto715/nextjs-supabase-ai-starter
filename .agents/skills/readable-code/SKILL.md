# リーダブルコードスキル

## 概要
このスキルは、「リーダブルコード」の原則を適用して、AIコーディングエージェントがクリーンで理解しやすいコードを書くのを支援します。新しいコードの作成、既存コードのリファクタリング、または可読性向上のためのコードレビュー時に使用してください。

## 核心原則
**コードは理解しやすくあるべき。** 基本的な目標は、他の誰か（または未来の自分）がコードを理解するのにかかる時間を最小限にすることです。

## このスキルを使用するタイミング
- 新しい関数、クラス、モジュールを書くとき
- 既存コードをリファクタリングするとき
- コードレビューと改善時
- デバッグやコード保守時
- 再利用可能なライブラリの作成時

## 主要な実践方法

### 1. 命名（名前に情報を詰め込む）

**具体的な単語を選ぶ**
- 悪い例: `get()`, `size()`, `stop()`
- 良い例: `fetch_page()`, `num_nodes()`, `kill()`
- 実際の操作を説明する正確な動詞を使用

**汎用的な名前を避ける**
- 避ける: `tmp`, `retval`, `foo`, `data`
- 例外: 狭いスコープ内の短命な変数は、一時的な保管が主目的なら`tmp`を使用可能
- 変数の目的や値を説明する名前を使用

**重要な情報を付加する**
- 単位を追加: `delay_secs`, `size_mb`, `max_kbps`, `degrees_cw`
- 属性を追加: `untrusted_url`, `html_utf8`, `plaintext_password`
- 重要な情報をエンコードするために接頭辞/接尾辞を使用

**誤解を避けるための明確な名前**
- 包括的な上限には`max_`と`min_`を使用
- 包括的範囲には`first`と`last`を使用
- 包括的/排他的範囲には`begin`と`end`を使用
- 真偽値には: `is_valid`, `has_permission`, `can_edit`, `should_retry`
- 否定形を避ける: `disable_ssl`より`use_ssl`を優先

**名前の長さのガイドライン**
- 短いスコープでは短い名前でOK
- 大きなスコープやクラスメンバーには長く説明的な名前
- 明確さのために簡潔さを犠牲にしない

### 2. コメント（何をコメントすべきか知る）

**コメントすべきでないもの**
- コード自体から素早く導き出せる事実
- 悪い名前―名前を修正すべき
- 関数名や変数名に既にある冗長な情報

**コメントすべきもの**
- ディレクターズコメンタリー: 洞察、根拠、トレードオフ
  ```python
  # 意外にも、このデータでは二分木がハッシュテーブルより40%速かった
  # ハッシュ計算のコストが左右比較よりも大きかったため
  ```

- 欠陥とTODO:
  ```python
  # TODO: より高速なアルゴリズムを使用
  # FIXME: JPEG形式のみ対応
  # HACK: API制限の回避策
  ```

- 定数の説明:
  ```python
  NUM_THREADS = 8  # num_processorsの2倍以上なら十分
  ```

- 予想される落とし穴:
  ```python
  # 外部サービスを呼び出す; 1分後にタイムアウト
  def send_email(to, subject, body):
  ```

- 高レベルの「全体像」の要約
- コーナーケースと入出力例

**コメントを正確かつコンパクトに**
- 曖昧な代名詞を避ける（it、this、that）
- 複雑な関数には入出力例を使用
- メカニクスではなく意図を述べる
- 情報密度の高い言葉を使用

### 3. コード構造

**美観は重要**
- 一貫したフォーマットとパターンを使用
- 役立つ場合は類似コードを列に揃える
- コードを論理的な段落に分割
- 意味のある順序を選び、一貫して使用

**制御フロー**
- 変化/検査される値を左に: `if (bytes_received < bytes_expected)`
- if/elseでは肯定的/単純/興味深いケースを最初に処理
- 最も単純なケース以外では三項演算子を避ける
- do/whileループを避け、whileループを優先
- ネストを減らすために早期returnを使用
- ネストレベルを最小化

**巨大な式を分解する**
- 部分式に説明変数を使用
- ド・モルガンの法則を適用してブール論理を簡素化
- 賢い「短絡論理」トリックを避ける
- 複雑な文を複数の単純な文に分解

### 4. 変数

**不要な変数を削除**
- 明確さを追加しない変数を削除
- 値を即座に処理して中間結果を削除
- 可能な場合は制御フロー変数を削除

**変数のスコープを縮小**
- 変数をできるだけ少ない行から見えるようにする
- 関数の先頭ではなく、使用直前で変数を定義
- 可能な場合はメンバー変数より局所変数を優先
- ブロックスコープを効果的に使用

**書き込み1回の変数を優先**
- 不変/const変数は推論しやすい
- 変数が変更される場所が多いほど、その値を追跡するのが難しい

### 5. コード組織化

**無関係なサブ問題を抽出**
1. コードの高レベル目標を特定
2. 無関係なサブ問題を解決している行を見つける
3. それらを別の関数に抽出

例:
```python
# 前: 関心事が混在
def find_closest_location(lat, lng, array):
    # ... 複雑な球面距離計算が検索ロジックと混在

# 後: サブ問題を抽出
def spherical_distance(lat1, lng1, lat2, lng2):
    # ... 独立した幾何学計算
    
def find_closest_location(lat, lng, array):
    # ... spherical_distance()を使用したクリーンなロジック
```

**一度に1つのタスク**
- コードが実行するすべてのタスクをリスト化
- 異なる関数や明確なセクションに分離
- 各セクションは単一の明確な目的を持つべき

**汎用コードを作成**
- 再利用可能なユーティリティを抽出
- ヘルパー関数のライブラリを構築
- 可能な限りプロジェクト固有の詳細から切り離す

## 実践的適用ステップ

### 新しいコードを書くとき:

1. **インターフェース設計から始める**
   - 明確で曖昧でない名前を選択
   - 目的とエッジケースを説明するコメントを書く
   - コードがどのように使用されるかを考慮

2. **まず平易な日本語で説明**
   - コードが何をすべきかをコメントとして書く
   - この説明を実装のガイドとして使用
   - キーワード/フレーズを変数名に抽出

3. **最小限で読みやすい実装を書く**
   - 複雑な式には説明変数を使用
   - 論理的なセクション/関数に分割
   - 各部分が1つのタスクを実行するようにする

4. **レビューとリファクタリング**
   - 不要な変数を削除
   - 制御フローを簡素化
   - 必要に応じて明確化コメントを追加

### 既存コードをリファクタリングするとき:

1. **現在のコードを読んで理解**
   - 高レベルで何をしているか特定
   - 混乱する部分を記録

2. **可読性の改善を適用**
   - 誤解を招く名前を修正
   - サブ問題を抽出
   - 式と制御フローを簡素化
   - 役立つコメントを追加

3. **動作が変わっていないことをテスト**
   - 機能を維持
   - 可読性/構造のみを改善

## 共通パターン

### パターン: 説明変数
```python
# 前
if line.split(':')[0].strip() == "root":
    ...

# 後
username = line.split(':')[0].strip()
if username == "root":
    ...
```

### パターン: 要約変数
```python
# 前
if request.user.id == document.owner_id:
    # ... 多数の行
if request.user.id != document.owner_id:
    # ... 多数の行

# 後
user_owns_document = (request.user.id == document.owner_id)
if user_owns_document:
    # ... 多数の行
if not user_owns_document:
    # ... 多数の行
```

### パターン: サブ問題の抽出
```python
# 前: 1つの関数に関心事が混在
def process_data(data):
    # 検証ロジック
    # 解析ロジック  
    # ビジネスロジック
    # フォーマットロジック

# 後: 関心事の分離
def validate_data(data): ...
def parse_data(data): ...
def apply_business_rules(data): ...
def format_output(data): ...

def process_data(data):
    validate_data(data)
    parsed = parse_data(data)
    result = apply_business_rules(parsed)
    return format_output(result)
```

### パターン: 早期return
```python
# 前: ネストした条件
def process(data):
    if data is not None:
        if data.is_valid():
            # ... メインロジック
        else:
            return error
    else:
        return error

# 後: 早期return
def process(data):
    if data is None:
        return error
    if not data.is_valid():
        return error
    # ... 自然なインデントレベルでメインロジック
```

## 避けるべきアンチパターン

- **賢いコード**: 賢さより明確さを優先
- **深いネスト**: 2-3レベル以上は追いにくくなる
- **大きな関数**: より小さく焦点を絞った関数に分割
- **曖昧な名前**: `data`, `info`, `process()`, `do_it()`
- **マジックナンバー**: 説明的なコメント付きの名前付き定数を使用
- **グローバル状態**: グローバル変数と副作用を最小化
- **冗長なコメント**: コードが明らかに行っていることを繰り返さない

## 覚えておくこと

- 良いコード > 悪いコード + 良いコメント
- コードを明確に説明できない場合、設計に何か問題がある可能性が高い
- 可読性の改善により、より良い設計が明らかになることが多い
- 最良のコードはコードがないこと―必要のない機能は実装しない
- 次にこのコードを保守する人が、あなたの住所を知っている凶暴なサイコパスだと思ってコードを書く

## 出力フォーマット

コードを改善する際は、以下を提供してください:
1. 見つかった可読性の問題の簡潔な説明
2. 適用した具体的な改善（命名、構造、コメントなど）
3. リファクタリングされたコード
4. 変更の主な利点

初めてコードを見る人にとって、すべての行が理解しやすくなることに焦点を当ててください。
